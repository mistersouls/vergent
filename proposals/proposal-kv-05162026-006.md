# Proposal: KV Read/Write Path

**Author**: Souleymane BA <soulsmister@gmail.com>
**Status:** Draft
**Date:** 2026-05-16
**Sequence:** 006
**Amends:** proposal-rebalance-11052026-005 (LMDB DBI rename), proposal-bootstrap-05022026-001 (contexts.toml schema)

---

## Summary

This proposal specifies the complete read/write path for Tourillon's key-value
layer: client-facing `tourctl kv put/get/delete` commands, coordinator fanout to
replicas, sloppy quorum with hinted handoff, storage layout redesign (DBI_KEYS /
DBI_LOG), read repair, and hint replay. It is the first proposal that makes
Tourillon a fully operational distributed KV store.

---

## Motivation

Proposals 001–005 established the ring, lifecycle FSM, gossip engine, and
partition rebalance. Data can now be physically distributed across nodes.
What is missing is the application layer that allows operators and clients to
actually read and write keys. Without it, the ring is an empty shell.

This proposal closes that gap: it defines every envelope kind, every coordinator
decision, every quorum rule, and every failure path that stands between a
`tourctl kv put` invocation and a durable, consistently-readable value.

## CLI contract

### `tourctl kv put`

```
tourctl kv put <key> <value> [-k <keyspace>] [-w/--quorum-write <n>]
               [-t/--type <type>] [--kt <type>] [--ks <type>] [--json]
```

Writes `value` under `key` in `keyspace` (default `"default"`). Returns exit
code 0 on quorum ack, non-zero on timeout or quorum failure.

#### Rendered output

```
$ tourctl kv put user:001 Alice
✓  user:001  written  [hlc 2026-05-17T10:23:01.123Z · #0001 · node-1, acks 2]
```

```
$ tourctl kv put user:001 Alice --json
{"hlc": {"wall_ms": 1747477381123, "counter": 1, "node_id": "node-1"}, "replicas": 2}
```

On quorum failure:
```
$ tourctl kv put user:001 Alice -w 3
✗  write failed: quorum unavailable (1/3 acks, timeout after 50 ms)
```

#### Envelope schema

**`kv.put`** (client → coordinator):
```
{
  key:          bytes,
  keyspace:     bytes,
  value:        bytes,   # msgpack-encoded value
  quorum_write: int      # optional; default = contexts.kv.quorum_write
}
```

**`kv.put.ok`** (coordinator → client):
```
{
  hlc:      {wall_ms: int, counter: int, node_id: str},
  replicas: int    # count of positive acks received (≥ quorum_write)
}
```

### `tourctl kv get`

```
tourctl kv get <key> [-k <keyspace>] [-r/--quorum-read <n>]
               [--kt/--key-type <type>] [--ks/--keyspace-type <type>] [--json]
```

Reads `key` from `keyspace`. Prints the confirmed version when available, or a
list of unconfirmed candidates in case 2 (uncertainty).

#### Rendered output

**Case 1 — confirmed version:**
```
$ tourctl kv get user:001
✓  confirmed
   key       user:001
   keyspace  default
   value     Alice
   type      str
   hlc       2026-05-17T10:23:01.123Z · #0001 · node-1
   replicas  2/2
```

**Case 1 — confirmed Tombstone (key deleted):**
```
$ tourctl kv get user:001
✓  confirmed  (deleted)
   key       user:001
   keyspace  default
   hlc       2026-05-17T10:24:15.001Z · #0001 · node-1
   replicas  2/2
```

**Case 2 — uncertainty (quorum reached but no version confirmed):**
```
$ tourctl kv get user:001
⚠  uncertain — 2 candidates, none confirmed  (quorum not reached at read time)
   #  value  type  hlc                           replicas
   1  Alice  str   2026-05-17T10:23:01.123Z…n1   1/2
   2  Bob    str   2026-05-17T10:22:59.807Z…n2   1/2
```

**Key not found (all responsive replicas returned absence):**
```
$ tourctl kv get user:999
✗  not found
```

**Case 3 — quorum unavailable (`kv.error`):**
```
$ tourctl kv get user:001 -r 3
✗  read failed: quorum unavailable (1/3 replicas responded)
```

**`--json` output (case 1):**
```
$ tourctl kv get user:001 --json
{
  "versions": [
    {
      "confirmed": true,
      "value": "Alice",
      "hlc": {"wall_ms": 1747477381123, "counter": 1, "node_id": "node-1"},
      "count": 2,
      "quorum_write": 2
    }
  ]
}
```

The `value` field in `--json` output is the msgpack-decoded Python value
serialised as JSON. Raw bytes that cannot be represented natively in JSON are
rendered as `{"_type": "bytes", "_value": "<base64>"}`.

#### Envelope schema

**`kv.get`** (client → coordinator):
```
{
  key:         bytes,
  keyspace:    bytes,
  quorum_read: int    # optional; default = contexts.kv.quorum_read
}
```

**`kv.get.ok`** (coordinator → client):
```
{
  versions: [
    {
      confirmed:    bool,
      value:        bytes | null,  # null for Tombstone; absent when not found
      hlc:          {wall_ms: int, counter: int, node_id: str} | null,
                                   # null when versions list is empty (not found)
      count:        int,           # replicas that returned this version
      quorum_write: int            # quorum_write stored in the Version
    },
    ...
  ]
}
```

An empty `versions` list signals key-not-found (all replicas returned absence).

### HLC display format

`HLCTimestamp` is `(wall_ms, counter, node_id)`. On the wire and in `--json`
output it is transmitted as a plain dict — the client is responsible for any
conversion to a datetime representation:

```
{"wall_ms": 1747477381123, "counter": 1, "node_id": "node-1"}   (wire / --json)
```

The CLI terminal display formats it from the dict fields as:

```
2026-05-17T10:23:01.123Z · #0001 · node-1   (human display, counter zero-padded to 4 hex digits)
```

**`kv.error`** (coordinator → client, all operations):

{

### Inter-node envelopes

These envelopes are never sent by `tourctl`; they travel only between nodes.

**`kv.replicate`** (coordinator → replica):
```
{
  key:          bytes,
  keyspace:     bytes,
  value:        bytes | null,  # null for Tombstone
  hlc:          {wall_ms: int, counter: int, node_id: str},
  quorum_write: int
}
```

**`kv.replicate.ok`** (replica → coordinator): empty payload `{}`.

**`kv.hint`** (coordinator → handoff node):
```
{
  key:          bytes,
  keyspace:     bytes,
  value:        bytes | null,
  hlc:          {wall_ms: int, counter: int, node_id: str},
  quorum_write: int,
  for_node:     str    # node_id of the primary that should eventually receive this
}
```

**`kv.hint.ok`** (handoff → coordinator): empty payload `{}`.

**`kv.fetch`** (coordinator → replica):
```
{
  key:      bytes,
  keyspace: bytes
}
```

**`kv.fetch.ok`** (replica → coordinator):
```
{
  found:        bool,
  value:        bytes | null,                                  # null when not found or Tombstone
  hlc:          {wall_ms: int, counter: int, node_id: str} | null,  # null when not found
  quorum_write: int | null                                     # null when not found
}
```

### Value encoding — inference and explicit types

`keyspace`, `key`, and `value` are all transmitted as bytes on the wire. The CLI
applies identical encoding rules to all three.

> The shell strips surrounding quotes before the CLI receives the argument.

#### Inference rule (no explicit type flag)

`json.loads(value)` is attempted. On success the resulting Python object is
encoded as msgpack. On failure the raw string is encoded as msgpack `str`.

| Shell value (after stripping) | `json.loads` | Wire encoding |
|-------------------------------|-------------|---------------|
| `Alice`   | ❌ | msgpack str   |
| `42`      | ✅ int | msgpack int |
| `3.14`    | ✅ float | msgpack float |
| `true`    | ✅ bool | msgpack bool  |
| `null`    | ✅ None | msgpack nil   |
| `[1,2,3]` | ✅ list | msgpack array |
| `{"a":1}` | ✅ dict | msgpack map   |

#### Explicit type flags (`-t`, `--kt`, `--ks`)

| Type    | Behaviour |
|---------|-----------|
| `str`   | Raw UTF-8, bypasses inference |
| `int`   | `int(value)` |
| `float` | `float(value)` |
| `bool`  | `true`/`1`/`yes` → `True`; `false`/`0`/`no` → `False` |
| `json`  | `json.loads(value)` forced |
| `bytes` | base64-decode → raw bytes |
| `null`  | msgpack nil; `<value>` is ignored |

**Priority**: `-t` / `--kt` / `--ks` > inference > fallback str.

#### `@` prefix — file read

An argument starting with `@` is interpreted as a file path; bytes are read raw.
`@` may be combined with `-t` / `--kt` / `--ks`; the file is read first, then
the type rule is applied.

Escaping a literal `@`: double it. `@@mentions` → literal `@mentions`.

```
@config.json          → read file, raw bytes
@config.json -t json  → read file, parse JSON
@@mentions            → literal "@mentions" (str by inference)
@@id -t int           → error: "@id" is not a valid integer
```

## Configuration

### `contexts.toml` additions

This proposal is the authority on the `[contexts.kv]` schema. Proposal-001
does not need to be revised for the schema itself; only its example file
must be updated (see section "Required amendments").

```toml
[contexts.kv]
quorum_write  = 2    # W default — acks required for a write
quorum_read   = 2    # R default — responses required for a read
```

**Precedence**: `CLI flag (-w/-r) > env var > contexts.toml > built-in default`
(W = 1, R = 1).

**Invariant R + W > RF**: the operator is responsible for choosing values
consistent with the cluster's `rf`. Tourillon does not enforce this at startup.
A misconfiguration produces frequent case-2 responses (uncertainty), never
silent corruption.

### `config.toml` additions — server-side `[kv]`

```toml
[kv]
fanout_timeout_ms             = 50    # default 50 ms — intra-DC p99 typically < 5 ms
                                       # minimum enforced: 10 ms (node refuses to start otherwise)
                                       # maximum: < 30 000 ms (transport RESPONSE_TIMEOUT)
hint_replay_stabilization_ms  = 2000  # delay before first replay after READY
hint_replay_concurrency        = 16   # concurrent in-flight hints per target
```

### Timeout coordinator vs. client timeout — two distinct things

The transport layer already defines `RESPONSE_TIMEOUT = 30.0 s`.
`TcpClient.request(env, timeout=...)` accepts this as a parameter.

| Role | Parameter | Value |
|------|-----------|-------|
| Coordinator → each replica | `TcpClient.request(timeout=fanout_timeout_ms/1000)` | from `config.toml → [kv] fanout_timeout_ms` |
| Client (`tourctl`) → coordinator | `TcpClient.request(timeout=RESPONSE_TIMEOUT)` | `30.0 s` constant |

**Ordering constraint**: `fanout_timeout_ms < RESPONSE_TIMEOUT × 1000` (i.e.
< 30 000 ms). The 50 ms default satisfies this comfortably.

When `TcpClient.request()` raises `ResponseTimeoutError` for a given replica,
the coordinator marks that replica non-responding and continues to wait for
others. `ConnectionClosedError` has the same semantics. The coordinator fails
(`kv.error`) only if quorum is not reached once all results are collected or the
global budget expires.

## Design

### Data model

#### `KvMetadata`

```python
@dataclass(frozen=True)
class KvMetadata:
    hlc: HLCTimestamp
    quorum_write: int = 1
```

`Version` and `Tombstone` replace their `metadata: HLCTimestamp` field with
`meta: KvMetadata`. Wire format: `quorum_write` is added as an optional field
(deserialised default `1`).

#### HLC uniqueness invariant

`HLCTimestamp` is defined as `(wall_ms, counter, node_id)`. This triplet is
globally unique by construction: two coordinators running at the exact same
physical millisecond will differ by `node_id`. All timestamp comparisons
**must** use the full triplet — never compare only `(wall_ms, counter)`. Any
code that truncates `node_id` from a comparison breaks the uniqueness guarantee
and must be considered a bug.

#### `Version` and `Tombstone`

Both record types carry `meta: KvMetadata` instead of a bare HLC. This is the
only structural change to existing record types.

#### `PreferenceEntry`

Already defined by proposal-002 / proposal-004. Used unmodified. The fields
`node_id`, `handoff`, `suspect`, and `readable` drive all routing decisions.

#### `PartitionHint` protocol

New protocol mirroring `PartitionStaging`, scoped to a specific `for_node_id`:

```python
class PartitionHint(Protocol):
    async def put(self, addr: StoreKey, val: bytes, meta: KvMetadata) -> None: ...
    async def delete(self, addr: StoreKey, meta: KvMetadata) -> None: ...
    async def mark_stale(self, addr: StoreKey, hlc: HLCTimestamp) -> None: ...
    def pending(self) -> AsyncIterator[Record]: ...
```

#### New methods on `PartitionStore`

```python
def hint(self, for_node_id: str) -> PartitionHint: ...

async def get(self, addr: StoreKey) -> Record | None: ...

async def put(self, addr: StoreKey, val: bytes, meta: KvMetadata) -> None: ...

async def delete(self, addr: StoreKey, meta: KvMetadata) -> None: ...

async def mark_phantom(self, addr: StoreKey, hlc: HLCTimestamp) -> None: ...
```

### Storage layout — DBI_KEYS and DBI_LOG

> ⚠️ This layout amends proposal-005. See section "Required amendments".

> **Implementation note**: LMDB is the reference storage engine and the
> concrete names (`DBI_KEYS`, `DBI_LOG`) are taken from it. The logical design
> — two key spaces, one ordered by `(pid, keyspace, key, hlc)` and one ordered
> by `(pid, hlc, keyspace, key)` — must be preserved by any storage engine that
> implements `PartitionStore`. An engine that does not use LMDB internally may
> name these structures differently, but must guarantee the same ordering
> semantics and atomicity properties.

The store uses two named key spaces with inverted roles relative to
proposal-005:

```
DBI_KEYS   key: pid (4B BE) | keyspace | key | hlc (12B)
           val: b"\x00"               — COMMITTED, visible to kv.get
               | b"\x01" + epoch (4B)  — STAGING (rebalance)
               | b"\x02" + node_id     — HINT (pending replay)
               | b"\x03"               — reserved (distributed lock, future)
               | b"\x04"               — STALE (hint replayed, pending GC)
               | b"\xff"               — PHANTOM (marked by read repair)

DBI_LOG    key: pid (4B BE) | hlc (12B) | keyspace | key
           val: <user bytes>           — empty for Tombstone
```

**DBI_KEYS** is ordered by `(pid, keyspace, key, hlc)` — a directory of all
versions of a key with their visibility tag. Optimised for key lookup.

**DBI_LOG** is ordered by `(pid, hlc)` — a chronological journal of all values
by partition. Optimised for temporal scan (rebalance).

**Atomicity**: all mutations write to both key spaces within a single storage
write transaction. All reads use a single storage read transaction covering
both key spaces — guaranteed consistent snapshot.

#### Visibility tag table

| Tag | Name | Written by | Visible to `kv.get` | GC eligible |
|-----|------|-----------|---------------------|-------------|
| `b"\x00"` | `COMMITTED` | coordinator (`kv.replicate`), rebalance `commit()` | ✅ | Future (version expiry) |
| `b"\x01" + epoch (4B)` | `STAGING` | rebalance `stage()` | ❌ | After `commit()` or `cleanup()` |
| `b"\x02" + node_id` | `HINT` | handler `kv.hint` | ✅ on handoff node | After replay → `STALE` |
| `b"\x03"` | `LOCK` *(reserved)* | future `kv.lock` | TBD | TBD |
| `b"\x04"` | `STALE` | `HintReplayManager` after replay | ❌ | GC / compaction |
| `b"\xff"` | `PHANTOM` | read repair (`mark_phantom()`) | ❌ | GC / compaction |

**Backward seek rule on a replica**: the seek is **scoped to a single key**.
Each entry in DBI_KEYS represents one specific version of a key — uniquely
identified by the composite storage key `(pid, keyspace, key, hlc)`. The
cursor is initialised at `(pid | keyspace | key | 0xFF×12)` (the highest
possible HLC for that key) and `prev()` steps backward through the versions
of that exact key in **descending HLC order** (newest first). The prefix
`(pid | keyspace | key)` acts as a hard guard: as soon as `prev()` yields an
entry whose key prefix no longer matches, the scan stops — we never cross into
a different `(keyspace, key)` combination. For each version entry encountered,
the tag byte determines the action:

```
b"\x00" (COMMITTED) → stop, return entry
b"\x02" (HINT)      → stop, return entry  — handoff node serves the data
b"\x01" (STAGING)   → skip, prev()
b"\x04" (STALE)     → skip, prev()
b"\xff" (PHANTOM)   → skip, prev()
prefix mismatch     → key absent, return None
```

The rule guarantees that a key whose only stored versions are STAGING, STALE,
or PHANTOM is treated as absent — a necessary invariant during rebalance and
after hint drain.

#### Local read algorithm — triggered by `kv.fetch`

```
1. DBI_KEYS backward seek on (pid | keyspace | key | 0xFF*12)
   → prev() → first entry with prefix (pid | keyspace | key)
2. tag == b"\x01", b"\x04", b"\xff" → skip, prev(), retry       [exceptional]
   tag == b"\x00" → COMMITTED, extract hlc from key
   tag == b"\x02" → HINT, extract hlc from key  (handoff node serves data)
3. DBI_LOG point lookup on (pid | hlc | keyspace | key) → user bytes
```

Step 2 stops on the first **visible** entry (`COMMITTED` or `HINT`) in practice:
**O(1) amortised**. Both accesses occur within the same storage read transaction.

#### Comparison with existing `PartitionStore` interfaces (proposal-005)

| Method | Before (proposal-005) | After |
|--------|-----------------------|-------|
| `scan()` | Walk `DBI_INDEX` (tags in val) + lookup `DBI_DATA` (value) | Walk `DBI_LOG` (value in val) + cross-lookup `DBI_KEYS` (tag) |
| `stage()` | Write `DBI_INDEX` (`\x01`) + `DBI_DATA` (value) | Write `DBI_KEYS` (`\x01`) + `DBI_LOG` (value) — identical semantics |
| `commit()` | Update `DBI_INDEX` `\x01→\x00` | Update `DBI_KEYS` `\x01→\x00` — identical |
| `cleanup()` | Delete `DBI_INDEX` + `DBI_DATA` | Delete `DBI_KEYS` + `DBI_LOG` — identical |
| `exists()` | Check `DBI_INDEX` for `\x01+epoch` | Check `DBI_KEYS` for `\x01+epoch` — identical |
| `last_staged_log_key()` | Max HLC in `DBI_INDEX` | Max HLC in `DBI_KEYS`, return `DBI_LOG` format `pid\|hlc\|ks\|key` — **same format** |
| `get()` *(new)* | N/A | `DBI_KEYS` backward seek + `DBI_LOG` lookup — 2 accesses |
| `mark_phantom()` *(new)* | N/A | Write `DBI_KEYS[pid\|ks\|key\|hlc] = \xff` — 1 write |

Network latency (≥ 1 ms per hop, multiple hops for quorum) dominates entirely
over the second storage access (~1–100 µs). The gap is measurable only in local
single-node benchmarks.

### Write path

#### Client → coordinator

The client sends `kv.put` to the KV endpoint defined in `contexts.toml`. The
receiving node becomes the **coordinator**; it validates inputs (key, value,
keyspace, `quorum_write`).

#### Preference list construction

The coordinator computes the `PartitionPlacement` for the key via `Partitioner`,
then calls `PlacementStrategy.preference_list(placement, topology,
probe_manager)`, which returns an ordered list of `rf` `PreferenceEntry` objects.
`SimplePreferenceStrategy` is the default.

**Primary selection rules (up to `rf` entries):**

| Member phase                 | Eligible as primary? | Handoff required? |
|------------------------------|----------------------|-------------------|
| `IDLE`, `JOINING`, `FAILED`  | ❌                    | N/A               |
| `READY` (not suspect)        | ✅                    | No                |
| `READY` (suspect)            | ✅                    | Yes               |
| `DRAINING`, `PAUSED`         | ✅                    | Always            |

**Handoff target selection**: a second clockwise traversal from the last primary
vnode finds handoff candidates — not `IDLE`/`JOINING`/`FAILED`, not already
used, not suspect. `PreferenceEntry.handoff` contains the handoff `node_id` or
`None` if no handoff is needed or no candidate is available. A given `node_id`
appears at most once across the entire preference list.

#### Fanout

The coordinator generates a unique `HLCTimestamp` for the request. `handoff` is
the sole routing signal:

- `entry.handoff is not None` → send `kv.hint` to `entry.handoff` with
  `entry.node_id` as the target primary for later replay.
- `entry.handoff is None` → send `kv.replicate` to `entry.node_id`, even if
  `entry.suspect=True` (no handoff candidate available — attempt anyway).
- If the coordinator is itself a `kv.replicate` target, it writes directly
  without going over the network.

All messages are sent **in parallel**.

#### Feedback to the failure detector

The coordinator notifies `ProbeManager` for every network operation, on both
the write path and the read fanout:

| Outcome | ProbeManager action | Notified target |
|---------|---------------------|----------------|
| Response received (ok or applicative) | `record_heartbeat(target)` | node actually contacted |
| `ResponseTimeoutError` | `record_miss(target)` | node actually contacted |
| `ConnectionClosedError` | `record_miss(target)` | node actually contacted |

The notified target is always the node actually contacted on the network:
`kv.replicate` → `entry.node_id`; `kv.hint` → `entry.handoff`; `kv.fetch` →
`entry.handoff` if present, else `entry.node_id`; hint replay → target `node_id`.

#### Structural limitation — failure detector detection gap

The preference list is calculated on the state known at time T. A node may be
de facto dead but still `suspect=False` (the FD has not yet crossed its
threshold — an unavoidable consequence of the Chandra-Toueg 1996 theorem). In
this case `entry.handoff is None` and the node does not respond. The coordinator
waits for the global timeout then returns `kv.error` if `quorum_write` is not
reached. This is accepted as a system limit; the correct response is to calibrate
the FD tightly to minimise this window.

#### Quorum and response

- The coordinator waits for **`quorum_write`** positive acknowledgements
  (`kv.replicate.ok` or `kv.hint.ok` — both count).
- If the threshold is not reached before the global timeout (`fanout_timeout_ms`
  in `config.toml → [kv]`), it returns `kv.error` to the client.
- Once the quorum is reached, it responds `kv.put.ok`.

### `Version` / `Tombstone` metadata additions

Each `Version` (and `Tombstone`) now stores:

**`quorum_write`**: the quorum used at write time. The oracle that allows the
coordinator to determine whether a version is **confirmed** without an external
commit log:

> A version V is **confirmed** if and only if at least `V.quorum_write` replicas
> return that same version during a read.

"Confirmed" is a property of the **read**, not the write: it means the version can
prove its quorum at read time. A write that acknowledged `quorum_write` replicas
may not be confirmed during a subsequent read if those replicas are temporarily
unavailable — the explicit trade-off of a leaderless system without a global
commit log.

Without this field the coordinator cannot distinguish a phantom write (reached a
single replica before coordinator crash) from a normal write where a replica is
stale.

**`replication_nodes`** *(future proposal, optional)*: list of `node_id` values
that actually received the data at write time — primaries and handoff combined.
Used in `kv.get` (case 2 only) to enable a targeted retry:

```
kv.get { key, keyspace, hint_nodes: [A, B, C, B_handoff] }
```

When `hint_nodes` is provided the coordinator bypasses the ring and queries the
listed nodes directly, after validating that each `node_id` belongs to the
current ring. Not part of this proposal; do not implement before the write and
read paths are stable.

### Read path

#### Client → coordinator

The client sends `kv.get` to the KV endpoint. The receiving node becomes the
**coordinator**; it validates inputs (key, keyspace, `quorum_read`).

#### Preference list construction

The same `PlacementStrategy.preference_list` is used as for writes. The full
list is evaluated — the coordinator discards no entry a priori.

#### Fanout

The coordinator sends `kv.fetch` **in parallel** to replicas (`kv.fetch` is
the inter-node envelope, distinct from `kv.get` which is the client→coordinator
contract):

- `entry.handoff is not None` → `kv.fetch` to `entry.handoff` only.
- `entry.handoff is None` and `entry.readable is True` → `kv.fetch` to
  `entry.node_id`.
- `entry.handoff is None` and `entry.readable is False` → entry is ignored.

Fanout never doubles requests: one entry produces at most one target. In normal
operation (`handoff=None` on all entries) read traffic is identical to a healthy
cluster.

The coordinator sends to all determined targets (not just `quorum_read` — to
maximise available responses). It waits for at least `quorum_read` responses
before deciding, or the global timeout.

If the coordinator is itself the fanout target, it reads directly from its local
store without going over the network.

#### Confirmed version determination

Each replica returns its most recent local version for the key (or an absence
indication). The coordinator groups responses by version and applies:

```
count(replicas that returned V) >= V.quorum_write  →  V is confirmed
```

**Why not the max HLC directly?** A replica may carry a version with a higher
HLC that never reached `quorum_write` (coordinator crashed mid-write). Returning
that HLC max would expose a phantom — a version the system was unable to confirm
from the read's perspective.

#### Response format — unified list

The `kv.get.ok` response is always a **list of versions**, whether or not there
is certainty. Each element carries its own `confirmed` flag. This unified format
avoids any parsing bifurcation on the client side.

```
{
  versions: [
    {
      confirmed: bool,
      value: <bytes> | null,   # null for a Tombstone
      hlc: <timestamp>,
      count: <n>,              # number of replicas returning this version
      quorum_write: <n>,       # quorum_write stored in the Version
    },
    ...
  ]
}
```

**Case 1 — confirmed version**: `versions` contains exactly one entry with
`confirmed: true`. It is the version with the highest HLC among those satisfying
`count >= quorum_write`. The client receives a reliable value.

**Case 2 — uncertainty**: the coordinator received at least `quorum_read`
responses but no version satisfies `count >= quorum_write`. `versions` contains
all candidates, each with `confirmed: false`. This indicates a temporarily
diverged state — typically a node that crashed after receiving a write the others
could not confirm, or an ongoing hint-drain window.

The client knows exactly why there is no certainty. It may choose to retry, wait
for cluster recovery, or apply its own policy. **None of the candidates is
presented as confirmed** — the client must not use these values as truth without
being aware of the situation.

**Case 3 — quorum unavailable** (`kv.error`): fewer than `quorum_read` replicas
responded before the global timeout. The coordinator does not have enough
information to produce a useful response.

#### Read repair

After determining the confirmed version V* (case 1 only), the coordinator
triggers a background `kv.replicate` towards every replica that returned a
version different from V* — including replicas that returned a **more recent
HLC** (phantom).

The receiving replica applies:

- If the received V* has HLC **≥** its local version → normal write in
  `DBI_KEYS` (`b"\x00"` COMMITTED) and `DBI_LOG` (value).
- If the received V* has HLC **<** its local version → the local version is a
  phantom. The replica calls `mark_phantom(hlc_local)`: an **atomic write
  transaction** that sets `\xff` in `DBI_KEYS` for the phantom's HLC **and**
  writes V* as `\x00` COMMITTED in the same transaction. `DBI_LOG` is unchanged for the
  phantom (value preserved for history, invisible to `kv.get`). Both the
  phantom mark and the V* write must commit together; a partial update is not
  acceptable.

**Atomicity invariant**: `mark_phantom` and the subsequent V* write are always
issued as a single storage transaction. The storage engine implementation is
responsible for guaranteeing this atomicity; the coordinator never issues them
as two separate calls.

Read repair is **best-effort and non-blocking**: its success or failure does not
delay the response to the client. The coordinator spawns repair tasks inside a
structured `asyncio.TaskGroup` that is not awaited before returning the
response to the client — repair tasks must not be orphaned. It is not
retransmitted on failure; the next read on the same key will trigger a new
repair if necessary. It is not triggered in case 2 (no confirmed reference
version).

### Consistency — R + W > RF rule

The classical distributed quorum invariant applies directly to Tourillon: if
the client configures `quorum_read` (R) and `quorum_write` (W) such that
`R + W > RF`, then every `kv.get` will see **at least one replica** that
participated in the last acknowledged `kv.put`, ensuring the returned confirmed
version is the most recent.

**Reference table (RF = 3):**

| W | R | R + W | > RF? | Guarantee |
|---|---|-------|-------|-----------|
| 1 | 1 | 2 | ❌ | None |
| 2 | 1 | 3 | ❌ | None |
| 1 | 2 | 3 | ❌ | None |
| 2 | 2 | 4 | ✅ | Strong consistency |
| 3 | 1 | 4 | ✅ | Strong consistency |
| 1 | 3 | 4 | ✅ | Strong consistency |
| 3 | 2 | 5 | ✅ | Maximum strong consistency |
| 2 | 3 | 5 | ✅ | Maximum strong consistency |
| 3 | 3 | 6 | ✅ | Maximum (zero availability if 1 node down) |

**Recommended defaults**: W = 2, R = 2 for RF = 3. Best
availability/consistency trade-off. Configurable in `contexts.toml`
(`quorum_write`, `quorum_read`) and overridable per request (`-w`, `-r`).

> **Sloppy quorum note.** With hinted handoff, a write may acknowledge W
> replicas including handoff nodes. The `R + W > RF` guarantee holds
> **if and only if the read uses the symmetric XOR rule** (fanout towards handoff
> when `entry.handoff is not None`). During a hint-drain window the read may not
> intersect the canonical write set and returns case 2 — honest and expected.

#### Two concurrent writes simultaneously confirmed *(not handled — future `kv.lock`)*

With RF = 3, W = 2: two concurrent clients can write V1 to `{A, B}` and V2 to
`{B, C}` simultaneously. Both acknowledge W = 2 replicas. Two concurrent reads
may obtain different results, both technically "confirmed". This is the direct
consequence of the absence of a leader and global commit log — the fundamental
trade-off of a leaderless system.

**Current behaviour**: the HLC breaks ties deterministically. `HLCTimestamp` is
defined as `(wall_ms, counter, node_id)` and is unique by construction — two
timestamps produced by different coordinators always compare unequal. The
coordinator retains the version with the highest HLC satisfying
`count >= quorum_write`. If `V2.hlc > V1.hlc` and B responds in both cases,
V2 is returned deterministically. The divergence window is bounded by read
repair convergence time.

**Future work**: a `kv.lock` envelope (tag `\x03`, reserved in `DBI_KEYS`) will
be introduced in a dedicated proposal to provide pessimistic locking for
clients that require strict write serialisation. Do not implement before the
write and read paths are stable.

---

### KV anti-entropy — FORBIDDEN

KV anti-entropy (periodic scan of all keys to reconcile replicas) is
**explicitly forbidden** in Tourillon. Without a distributed commit log, any
background scanner that propagates the version with the highest HLC can promote
a phantom to a confirmed version and irrecoverably overwrite valid data. Read
repair is the only safe reconciliation path because it operates **after**
establishing consensus via quorum.

### Hint replay

#### Overview

When a node `node_id` returns to `READY` after a period of suspicion, `PAUSED`,
or `DRAINING`, handoff nodes that stored hints for it (`DBI_KEYS` tag
`b"\x02" + node_id`) must replay those hints. The `HintReplayManager` is the
component responsible for this transfer on each node.

#### Trigger — start condition

The only condition for starting a replay session is gossip:

```
phase(node_id) == READY   (MemberRegistry)
```

The local probe (`ProbeManager`) does not intervene in the start decision — the
replay is itself an active liveness signal (see "Feedback to the failure
detector" below).

**Stabilisation**: even when the condition is satisfied, `HintReplayManager`
waits `hint_replay_stabilization_ms` (default 2 000 ms) before starting, to
filter gossip flapping. If the phase becomes ≠ `READY` during the window, the
timer is cancelled.

```
phase == READY
    → start timer(stabilisation_ms)
    → if phase ≠ READY before expiry → cancel timer
    → at expiry → start ReplaySession(node_id)
```

During an active session: if the gossip phase transitions to ≠ `READY`, the
session is abandoned immediately. Unplayed hints retain their `\x02` tag and
will be replayed in the next session.

#### Feedback to the failure detector

`HintReplayManager` notifies `ProbeManager` on every request result:

| `TcpClient.request()` result | ProbeManager action |
|------------------------------|---------------------|
| `kv.replicate.ok` (success) | `record_heartbeat(node_id)` |
| `ResponseTimeoutError` | `record_miss(node_id)` |
| `ConnectionClosedError` | `record_miss(node_id)` |

Gossip heartbeats are passive low-rate probes. Hint replay is active application
traffic — a timeout on `kv.replicate` is a strong signal of unreachability, more
reliable than a missed heartbeat. These notifications feed the FD φ:

- Repeated successes → φ decreases → probe `LIVE` → gossip confirms `READY` →
  node stays in preference list without handoff.
- Repeated failures → φ increases → probe becomes `SUSPECT` →
  `SimplePreferenceStrategy` adds a handoff → new writes are protected while
  replay continues (with backoff).
- φ exceeds `FAILED` threshold → gossip propagates the transition → replay
  session is abandoned (phase ≠ `READY`).

`HintReplayManager` never reads probe state to decide whether to send — it sends
and notifies. The failure detector draws its own conclusions.

#### Replay protocol

`HintReplayManager` opens a `TcpClient` connection to `node_id` and sends hints
one by one in ascending HLC order (natural `DBI_LOG` order). Each hint is sent
as a regular `kv.replicate` — the receiver makes no distinction between a hint
replay and ordinary replication.

**Sequence per hint:**

```
1. Read hint (keyspace, key, hlc, value) from DBI_LOG
   + confirm tag \x02 in DBI_KEYS
2. Send kv.replicate → node_id (TcpClient.request, timeout=fanout_timeout_ms)
3. Receive kv.replicate.ok
4. Storage write transaction:
       UPDATE DBI_KEYS[pid|keyspace|key|hlc] = b"\x04"   ← HINT → STALE
   → DBI_LOG unchanged; physical cleanup delegated to future GC/compaction
```

After the tag update the entry is `STALE`: invisible to `kv.get` (backward seek
skips it), non-replayable (tag `\x04` ≠ `\x02`).

**Bounded concurrency**: at most `hint_replay_concurrency` hints in flight
simultaneously towards the same `node_id` (default 16). Controlled by an
`asyncio.Semaphore`.

#### Error handling and retry

| Error | Behaviour |
|-------|-----------|
| `ResponseTimeoutError` or `ConnectionClosedError` | Session abandoned, retry after exponential backoff (base 1 s, max 60 s, ±20 % jitter) |
| `node_id` transitions ≠ `READY` during replay | Session abandoned immediately; hints preserved |
| Handoff node itself crashes | On restart, re-scan `DBI_KEYS` for `\x02+node_id`; any partial session resumes from the beginning (idempotent) |
| Ring change invalidates `node_id` as owner | Apply DRAINING algorithm (see below) |

#### Hint lifetime — no TTL

Hints have **no TTL**. They persist until one of two events:

- **Hint replay**: primary returns to `READY`, `HintReplayManager` replays hints
  → tag `\x04` (`STALE`).
- **Ring change / DRAINING**: operator triggers `DRAINING` on the primary →
  rebalance redistributes partitions → hints become stale (see below). Tag
  updated to `\x04` without replay.

A TTL would create silent data loss: if the primary returns after expiry, it
never receives the evicted hints. This scenario is undetectable without a full
audit log. The correct response to a long-absent primary is `tourctl node drain`
— not an application timer.

#### DRAINING of the primary — hint redirection

When `node_id` enters `DRAINING`, stored hints for it cannot be replayed to it.
They must be redirected to its **replacement** in the ring.

**Replacement = first `READY` non-suspect node in the current preference list
for the hint's partition**, other than `node_id` itself and other than the
current handoff node (which already holds the data).

**Algorithm:**

```
For each hint (keyspace, key, hlc) with tag \x02+node_id
    if node_id.phase == DRAINING:
        placement = Partitioner.placement(key, keyspace)
        pref_list = PlacementStrategy.preference_list(placement, topology, probe_manager)
        replacement = first entry in pref_list such that:
            entry.node_id ≠ node_id
            AND entry.handoff is None          # READY, not suspect
            AND entry.node_id ≠ self.node_id   # not the handoff node itself
        if replacement found:
            send kv.replicate → replacement
            → ack → UPDATE DBI_KEYS tag \x02+node_id → \x04 (STALE)
        else:
            keep hint (\x02), retry later
```

**Deduplication**: the replacement may already hold the data if it was in the
original replication set. It accepts it without effect (same HLC, upsert
idempotent). Future: use `replication_nodes` in `Version` to skip the send
if the replacement is already listed.

**No silent discard**: if no `READY` non-suspect replacement is available, the
hint is kept. The session enters backoff and retries when topology improves.

#### Ring change during replay

After a ring change a handoff node may find that `node_id` has left the ring
(tokens removed, different epoch). The rebalance that always precedes a ring
change redistributes data that `node_id` owned — but not hints on the handoff
node (data that `node_id` never received). These hints are not covered by the
rebalance.

Behaviour: apply the DRAINING algorithm — find the first `READY` non-suspect
replacement in the current preference list and replay towards it. The hint is
never silently evicted.

`HintReplayManager` checks after each ring change whether ongoing sessions
target `node_id` values still active in the ring, and switches to DRAINING mode
for those that have left.

#### Worst case — massive hint accumulation

If a primary node is absent for a long time on an active cluster, a handoff node
can accumulate hundreds of thousands of hints. On primary return, replaying `N`
hints takes `N × (RTT + storage latency)` ms sequentially, attenuated by bounded
concurrency (`hint_replay_concurrency = 16`).

There is no inter-hint compaction mechanism (e.g. keep only the latest HLC per
key) to avoid introducing merge logic in the handoff. Each hint is a history
entry, potentially visible in future `kv.get` responses (case 2).

`kv.fetch` is the inter-node counterpart of `kv.get`. A replica receiving

## Design decisions

### Decision: unified response list for `kv.get`

**Alternatives considered**: separate response envelopes for case 1 and case 2;
a single "winner" value with a `uncertain: bool` flag.

**Chosen because**: a list with per-entry `confirmed` flags gives the client a
complete, unambiguous picture without requiring different parsing logic per case.
The coordinator never silently hides candidates from the client.

### Decision: `quorum_write` embedded in `Version`

**Alternatives considered**: maintain a separate quorum log per key; always use
the cluster's current `quorum_write` at read time.

**Chosen because**: the cluster's `quorum_write` may change between write and
read. Embedding the write-time quorum in each version lets the coordinator apply
the exact threshold used when the version was produced, making confirmation
deterministic regardless of subsequent reconfiguration.

### Decision: no hint TTL

**Alternatives considered**: expire hints after a configurable duration;
automatically drain when accumulation exceeds a threshold.

**Chosen because**: a TTL creates silent data loss that is undetectable without
a global audit log. The operational lever for a long-absent primary is
`tourctl node drain`, not a timer.

### Decision: KV anti-entropy forbidden

**Alternatives considered**: bounded anti-entropy scans on idle partitions.

**Chosen because**: without a distributed commit log, any background scanner
that applies HLC-max wins risks promoting phantoms to confirmed status. Read
repair is the only safe reconciliation path.

### Decision: read fanout to all eligible targets

**Alternatives considered**: stop fanout once `quorum_read` responses arrive;
send to `rf` targets always.

**Chosen because**: sending to all eligible targets maximises the chance of
collecting enough responses before the global timeout, especially in degraded
scenarios. The cost is bounded by `rf` and is negligible compared to network
latency.

### Decision: DBI_KEYS / DBI_LOG layout vs. proposal-005

**Alternatives considered**: keep proposal-005's DBI_INDEX / DBI_DATA layout.

**Chosen because**: the KV `get` operation needs an O(1) amortised lookup by
`(keyspace, key)` that directly yields the most recent visible version. With the
proposal-005 layout the value (user bytes) was stored in the key-ordered key
space, requiring a secondary scan to find the tag. Swapping roles places the tag
in the key-ordered key space (one seek + one point lookup) and the value in the
HLC-ordered key space (natural scan order for rebalance). This also matches the
rebalance scan pattern: walk DBI_LOG chronologically and cross-lookup DBI_KEYS
for the tag. The layout names (`DBI_KEYS`, `DBI_LOG`) are LMDB-specific but the
logical structure is storage-engine agnostic.
