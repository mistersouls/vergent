# Security

## Threat Model

Tourillon assumes hostile networks and potentially compromised clients. The baseline objective is confidentiality, integrity, peer authenticity, and auditability.

Primary threats:
- Passive traffic interception.
- Active man-in-the-middle attacks.
- Unauthorized node enrollment.
- Replay of valid messages.

## Mandatory mTLS Requirements

- All communications MUST require mutual TLS.
- Nodes MUST validate peer certificate chains and identities before processing payloads.
- Clients MUST present trusted certificates when connecting to any node.
- Cleartext fallback MUST NOT exist.

## Certificate Generation Workflow

Tourillon provides `tourillon pki` commands to bootstrap the certificate
hierarchy required for mTLS. The workflow follows a standard CA → leaf
certificate chain.

The `cryptography` package ships as a mandatory Tourillon runtime dependency. No
extra install step is required — `uv sync` or `pip install tourillon` makes all
`tourillon pki` commands available immediately.

**Step 1 — Generate the Certificate Authority**

```bash
tourillon pki ca --common-name "Tourillon CA" --out-dir ./pki --days 3650
```

This produces a self-signed CA certificate and private key. The CA key file is
always written with mode 0600 (or the Windows ACL equivalent). The operator
is responsible for storing the CA private key offline; it must never be placed
on cluster nodes.

**Step 2 — Issue server certificates**

Each cluster node requires a server certificate signed by the CA. The
`tourillon pki server` command requires at least one Subject Alternative Name
(`--san-dns` or `--san-ip`). A bare CN-only server certificate is rejected
because modern TLS clients do not use the CN field for hostname verification.

**Step 3 — Issue client certificates**

Applications and administrative clients that connect to cluster nodes over mTLS
require a client certificate signed by the same CA.

**Private key handling**

- All private key files generated by the CLI are written with mode 0600.
- Private key material is never written to log output, regardless of the
  verbosity level.
- The CLI will refuse to overwrite an existing key file unless `--force` is
  passed, preventing accidental key replacement.

**Certificate rotation**

Certificate rotation support (without cluster downtime) is planned as part of
Milestone 3 and will be exposed via additional `tourillon pki` sub-commands.
Until then, operators must coordinate a rolling restart to deploy new
certificates.

- Trust roots MUST be explicitly configured.
- Certificate rotation SHOULD be supported without full cluster downtime.
- Revocation strategy MUST be defined (CRL, OCSP, or short-lived cert policy).
- Enrollment flows for new nodes MUST include identity verification.

## AuthN/AuthZ Boundaries

- Authentication MUST be certificate-based at transport level.
- Authorization SHOULD define role boundaries (client vs node, read vs write, admin operations).
- Sensitive admin endpoints MUST be separately scoped and audited.

## Security Hardening Baseline

- TLS versions/ciphers MUST follow modern secure defaults.
- Private keys MUST be protected at rest and never logged.
- Security-relevant events MUST be logged with correlation IDs.
- Replay protection SHOULD combine nonce/timestamp/session constraints where applicable.

## Security Acceptance Criteria

- Nodes refuse non-mTLS connections.
- Invalid or expired certificates are rejected deterministically.
- Certificate rotation procedure is tested in staging.
- Audit logs allow reconstruction of security incidents.
